# Paid Readiness Baseline (Phase 2)

## Purpose

Define what "premium-ready" means for the plugin-first core before device/export scope expands.

## Paid Value Hypothesis

Users may pay for a focused, high-quality in-vault reader workflow when it is:

- faster than manual reading for real notes,
- stable across daily use,
- differentiated by polish and UX quality.

## Baseline Criteria (Required)

1. In-context experience:
   Reader opens as a side panel, not a disruptive external workflow.
2. Distinctive interface:
   Framed shell UI with strong readability defaults and clean controls.
3. Personalization:
   Persistent settings for speed, ORP, typography, and accent theme.
4. Reliability:
   Startup recovery and deterministic error handling are tested and stable.
5. Discoverability:
   Command and usage flow are documented clearly in README and docs.

## Pricing Readiness Inputs (Post-Phase-2)

- Activation signal: repeated weekly usage by at least a small beta cohort.
- Retention signal: users return after first-week install.
- Conversion signal: users report clear time-saving or focus benefits.

## Explicit Non-Goals For Phase 2

- Device flash/write automation.
- Firmware forks or custom board protocols.
- Hosted backend services.
- Subscription/payment infrastructure implementation.
- BYOK AI expansion beyond existing plan placeholder scope.

## Exit Decision

If Phase 2 baseline is met and validation signals are positive, choose one of:

- free + portfolio-led distribution first, or
- paid-tier packaging with a free core path.
