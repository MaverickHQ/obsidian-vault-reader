# Design

## Document Info

- Status: Active
- Depth: Level 400
- Scope: Core architecture for the Obsidian-first pivot
- Last Updated: 2026-05-22

## 1. Architectural Thesis

The project should be designed as a publishable Obsidian plugin first, with RSVP Nano interoperability as an adapter layer. The plugin owns reading, normalization, settings, and export decisions. The device firmware remains upstream-owned for v1.

This yields four important benefits:

1. The product can ship before any firmware changes exist.
2. The repo can be shaped for Obsidian community plugin submission.
3. Compatibility work can be validated against a real device early.
4. BYOK AI can be added without infecting the core reading workflow.

## 2. Design Principles

- **Plugin first**: The plugin is the product and should be installable on its own merits.
- **Interop over reinvention**: Match the current RSVP Nano workflow before proposing new firmware or protocols.
- **Local first**: Core reading and export work with no account and no backend.
- **Behavior-first TDD**: Tests target public behavior, not internal implementation trivia.
- **Repo honesty**: The active docs describe only what we intend to build now.

## 3. Recommended Repo Shape

For community plugin publication, the long-term target is a plugin-first repo layout at the repository root:

```text
/
├── manifest.json
├── package.json
├── versions.json
├── src/
├── styles.css
├── tests/
├── fixtures/
├── docs/
└── README.md
```

### 3.1 Current State

Historical clone leftovers from the old scope were removed so the repository contains only the active plugin implementation path.

### 3.2 Near-Term Decision

Before implementation begins in earnest, we should flatten the publishable plugin into the repo root or make a dedicated plugin repo. The former is preferable because it keeps the GitHub portfolio story simple and aligns with Obsidian's release expectations.

## 4. System Components

### 4.1 Core Plugin Components

| Component | Responsibility |
|-----------|----------------|
| `PluginShell` | Obsidian lifecycle, commands, settings registration, view registration |
| `SourceResolver` | Reads active note or editor selection from Obsidian |
| `MarkdownNormalizer` | Converts note content into stable reading text |
| `Tokenizer` | Splits normalized text into display tokens |
| `ORPEngine` | Computes focus point metadata for tokens |
| `TimingEngine` | Controls word pacing and pause rules |
| `ReaderController` | Session lifecycle, state transitions, position persistence |
| `ReaderView` | Visual rendering, controls, keyboard handling |
| `ExportAdapter` | Serializes reading text into RSVP Nano-compatible output |
| `TargetWriter` | Writes artifacts into a chosen folder or SD card layout |
| `AIAdapter` | Optional provider abstraction for BYOK commands |

### 4.2 Supporting Assets

| Asset | Purpose |
|-------|---------|
| `fixtures/notes` | Stable sample notes for manual and automated validation |
| `fixtures/exports` | Known-good export outputs used for compatibility regression checks |
| `docs/adr` | Decision records for major architectural choices |

## 5. Core Data Flow

### 5.1 Read Inside Obsidian

```text
Obsidian editor/file
  -> SourceResolver
  -> MarkdownNormalizer
  -> Tokenizer
  -> ORPEngine + TimingEngine
  -> ReaderController
  -> ReaderView
```

### 5.2 Export For RSVP Nano

```text
Obsidian editor/file
  -> SourceResolver
  -> MarkdownNormalizer
  -> ExportAdapter
  -> RSVP Nano-compatible artifact
  -> TargetWriter
  -> SD card or chosen output folder
```

### 5.3 Optional AI Enhancement

```text
Obsidian editor/file
  -> SourceResolver
  -> MarkdownNormalizer
  -> AIAdapter
  -> Preview step
  -> ReaderView and/or ExportAdapter
```

## 6. Reading Engine Design

### 6.1 Session State

The reader state machine should remain intentionally small:

- `IDLE`
- `READY`
- `PLAYING`
- `PAUSED`
- `COMPLETE`
- `ERROR`

State transitions must be explicit and covered by tests.

### 6.2 Normalization Boundary

The `MarkdownNormalizer` is the most important boundary in the system. It must:

- strip or flatten Markdown formatting,
- preserve headings when they help reading flow,
- remove noise such as link syntax and repeated formatting marks,
- avoid surprising the user by inventing new prose,
- produce deterministic output for the same input.

This module should be tested with fixtures before the reader UI is considered stable.

### 6.3 Display Model

The reader should treat each token as a view model:

```ts
interface DisplayToken {
  raw: string;
  spoken: string;
  orpIndex: number;
  delayMs: number;
  position: number;
}
```

That keeps ORP and timing logic testable without requiring the DOM.

## 7. RSVP Nano Interoperability Design

### 7.1 Strategy

The plugin should target the existing RSVP Nano workflow rather than inventing a new device abstraction.

For v1 this means:

- follow the current `.rsvp` artifact expectations,
- follow the current SD card folder layout,
- validate on a real device,
- keep the export layer isolated so it can evolve if upstream changes.

### 7.2 Compatibility Contract

The compatibility contract is:

1. The plugin generates files that the current RSVP Nano firmware can open.
2. The plugin writes them into the correct library path.
3. The project keeps fixtures from real successful exports to detect regressions.

### 7.3 Fallback Philosophy

If exact serializer behavior is difficult to reproduce initially, we should still preserve the architecture boundary:

- first understand upstream format and conversion expectations,
- then mirror the format as directly as possible,
- only fall back to indirect workflows if that becomes necessary.

The key rule is that we do not invent a second export story while claiming compatibility.

## 8. BYOK AI Design

### 8.1 Provider Interface

AI should sit behind a narrow interface:

```ts
interface AIProvider {
  summarize(input: string): Promise<string>;
  rewriteForReading(input: string): Promise<string>;
  extractKeyTerms?(input: string): Promise<string[]>;
}
```

### 8.2 Rules

- No hosted project backend.
- No background sync of note contents.
- No silent rewriting of vault files.
- No AI-coupled reader architecture.

### 8.3 UX Pattern

Each AI command should follow:

1. User invokes command.
2. Provider runs on normalized text.
3. Result is previewed.
4. User explicitly chooses one of:
   - read result now,
   - export result,
   - discard result.

## 9. Testing Strategy

### 9.1 Test Layers

| Layer | Purpose |
|-------|---------|
| Unit | Tokenization, ORP, timing, normalization, serializer behavior |
| Integration | Obsidian API interaction, settings persistence, file writing |
| Fixture | Exact normalized outputs and export artifacts |
| Manual Obsidian | Desktop UX validation with a real vault |
| Manual Device | Real hardware validation with flashed RSVP Nano firmware |

### 9.2 Local Obsidian Validation Loop

The local development loop should look like this:

1. Use a dedicated test vault.
2. Build the plugin in watch mode.
3. Install the plugin into the vault's `.obsidian/plugins/<plugin-id>` directory.
4. Reload Obsidian after changes.
5. Run a fixed set of fixture notes through the reader and export commands.

This loop should be documented in the README once the scaffold exists.

### 9.3 Device Validation Loop

The device validation loop should look like this:

1. Flash the device with the current RSVP Nano firmware.
2. Prepare a FAT32 SD card with the expected folder layout.
3. Export fixture notes from the plugin to `books` and `articles`.
4. Copy files to the card.
5. Insert the card and confirm the device opens the files without manual repair.

We should not treat device validation as optional. It is the proof that the interop story is real.

## 10. Community Plugin Release Design

To publish on the Obsidian community plugin directory, the repo must eventually satisfy these release conditions:

- root `README.md`,
- root `LICENSE`,
- root `manifest.json`,
- GitHub releases with `main.js`, `manifest.json`, and optional `styles.css`,
- `versions.json` for compatibility tracking.

The release process therefore influences repo layout early. It is not just a packaging concern.

## 11. Risks And Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| Repo remains monorepo-shaped | Harder publishing and weaker portfolio story | Flatten to plugin-first layout before implementation |
| Upstream RSVP Nano format changes | Export regressions | Keep fixtures from successful device reads and isolate export adapter |
| Markdown normalization is too aggressive | User trust loss | Fixture-based normalization tests and preview-friendly design |
| AI becomes product center of gravity | Scope creep and quality drop | Make AI optional and adapter-based |
| Manual device steps are flaky | Interop story feels fake | Standardize fixture notes and SD card validation checklist |
