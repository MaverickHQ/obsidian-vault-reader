# Project Structure

## Purpose

This document explains the active repository layout for public reviewers, future contributors, and release preparation. Vault Reader is structured as a root-level Obsidian plugin repo so GitHub releases and Obsidian community submission can use the standard plugin assets directly.

## Release-Critical Root Files

| Path                | Purpose                                                                   |
| ------------------- | ------------------------------------------------------------------------- |
| `manifest.json`     | Obsidian plugin metadata used by local installs and community submission. |
| `versions.json`     | Obsidian compatibility map for released plugin versions.                  |
| `package.json`      | npm scripts, package metadata, and dependency list.                       |
| `package-lock.json` | Locked dependency graph for reproducible installs.                        |
| `styles.css`        | Runtime plugin styling shipped with releases.                             |
| `README.md`         | Public product overview, installation, usage, privacy, and limitations.   |
| `LICENSE`           | MIT license for the public project.                                       |
| `SECURITY.md`       | Security reporting and supported-version policy.                          |
| `CONTRIBUTING.md`   | TDD-first contribution and validation guidance.                           |

`main.js` is generated by the build and must not be tracked.

## Source Code

| Path                    | Purpose                                                                                             |
| ----------------------- | --------------------------------------------------------------------------------------------------- |
| `src/main.ts`           | Obsidian plugin entrypoint.                                                                         |
| `src/plugin-startup.ts` | Lifecycle, commands, views, and settings orchestration.                                             |
| `src/reader/`           | Reading source resolution, normalization, tokenization, timing, controller, view, and highlighting. |
| `src/settings/`         | Plugin settings, persistence, and settings tab behavior.                                            |
| `src/errors/`           | User-facing error normalization.                                                                    |
| `src/ui/`               | Shared UI copy and small view helpers.                                                              |

## Tests

| Path                         | Purpose                                                                                      |
| ---------------------------- | -------------------------------------------------------------------------------------------- |
| `tests/unit/`                | Behavior and policy tests for domain logic, metadata, docs, security, and release contracts. |
| `tests/integration/`         | Obsidian-adjacent behavior tested with mocks and runtime harnesses.                          |
| `tests/e2e/`                 | Release package and clean install smoke tests.                                               |
| `tests/fixtures/`            | Active fixture regression tests for reader-relevant Markdown behavior.                       |
| `tests/helpers/`             | Test harness utilities.                                                                      |
| `tests/mocks/`               | Obsidian API mocks used by automated tests.                                                  |
| `tests/no-secrets.sh`        | Secret scanning guard for tracked project files.                                             |
| `tests/repo-shape.sh`        | Repository structure and forbidden artifact guard.                                           |
| `tests/release-artifacts.sh` | Release package validation guard.                                                            |

Deferred hardware/export tests are archived instead of running in the active suite.

## Fixtures

| Path                            | Purpose                                                                                          |
| ------------------------------- | ------------------------------------------------------------------------------------------------ |
| `fixtures/notes/`               | Source Markdown samples used by automated normalization tests.                                   |
| `fixtures/expected/normalized/` | Expected normalized text snapshots.                                                              |
| `fixtures/manual-test-vault/`   | Local Obsidian vault used for manual QA and local deployment.                                    |
| `fixtures/exports/`             | Placeholder for future generated export samples; active release packaging must not depend on it. |
| `fixtures/archive/`             | Ignored local-only scratch archive for manual vault cleanup.                                     |

Manual test vault scratch notes, deployed plugin assets, workspace state, `.base`, and `.canvas` files are ignored unless explicitly whitelisted.

## Scripts

| Path                                       | Purpose                                                |
| ------------------------------------------ | ------------------------------------------------------ |
| `scripts/deploy-local-vault.mjs`           | Copies built plugin assets into the manual test vault. |
| `scripts/dev-local-vault.mjs`              | Starts local development deployment workflow.          |
| `scripts/export-clean-tree.mjs`            | Exports a no-history public source tree from Git.      |
| `scripts/verify-clean-tree.mjs`            | Verifies the clean export in an isolated scratch copy. |
| `scripts/clean-tree-policy.mjs`            | Shared clean public tree validation rules.             |
| `scripts/package-release.mjs`              | Builds versioned release artifacts.                    |
| `scripts/release-package-policy.mjs`       | Shared release package validation rules.               |
| `scripts/release-candidate-validation.mjs` | Clean-vault release install validation helpers.        |

Use `npm run export:clean-tree` to create `release/clean-public-tree` from Git-tracked source files without `.git`, generated release assets, local vault deployment output, or ignored scratch files.

Use `npm run verify:clean-tree` before creating the public GitHub repository. It exports the clean tree, installs dependencies inside the export, and runs the standard verification gate there.

Use `npm run verify:release` before clean public repo export and before creating a GitHub Release candidate. It composes formatting, linting, tests, release build checks, release packaging, release E2E smoke, audit, and no-secrets validation.

## Generated And Local-Only Files

These are expected locally but must not be tracked:

- `node_modules/`
- `main.js`
- `dist/`
- `build/`
- `.vite/`
- `coverage/`
- `release/`
- `.DS_Store`
- `.env` and `.env.*`
- manual test vault workspace files
- deployed plugin assets under `fixtures/manual-test-vault/.obsidian/plugins/vault-reader/`
- local scratch material under `fixtures/archive/`

## Archive Policy

Archive files are kept when they explain project history or preserve deferred work. They should not be referenced as active release requirements unless a later phase explicitly reactivates them through requirements, design, ADR, tasks, and tests.

Tracked archives belong under `docs/archive/`. Local scratch archives belong under ignored `fixtures/archive/`.
